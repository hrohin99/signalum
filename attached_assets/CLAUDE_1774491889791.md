## What is Signalum?

Signalum is a **multi-tenant competitive intelligence SaaS platform** built by Rohin (Product Manager at Entrust). It is live at `watchloom.rohin.co` (migrating to `signalum.rohin.co`). The app tracks competitors, regulations, and industry topics for a given workspace, using Perplexity for ambient intelligence and Claude for AI-generated summaries and strategic analysis.

Rohin is **non-technical** and works by having Claude diagnose issues and write precise prompts for **Replit AI** to execute.

## Critical Infrastructure Rules

1. **NEVER run `drizzle-kit push`** — will drop ~25 columns
2. **Always use `db.execute(sql`...`)` NOT `pool.query`** for INSERT/UPDATE in new code
3. **New DB columns** → always add to `server/dbSafety.ts` with `ADD COLUMN IF NOT EXISTS`
4. **DB connection** → `DATABASE_URL` with `?ssl` suffix, unified dev+prod via Replit Secrets
5. **Never use "Copy dev to production"** in Replit — wipes production data
6. **Before any deploy** → check migration preview for DROP statements, cancel if found
7. **Replit deployment fix** → source edit → `npm run build` → Republish (only reliable path)
8. **`staleTime: Infinity`** is set globally in `queryClient.ts` — always add `staleTime: 0, gcTime: 0, refetchOnMount: "always"` to individual queries that need fresh data
9. **Replit shell vs production server may use different DB connections** — always verify data through the browser/API, not just shell queries
10. **Drizzle `sql` template cannot pass JS arrays** — use `sql.join(ids.map(id => sql`${id}::uuid`), sql`, `)` for IN clauses

## Database Schema Summary

### Core tables
- **`workspaces`** — one per user, contains `categories` JSONB column with all entities
- **`user_profiles`** — role, onboarding context, digest settings
- **`captures`** — intel items matched to entities

### Competitive Dimensions (NEW — March 2026)
- **`competitive_dimensions`** — dimension groups per workspace
  - `id` (uuid PK), `workspace_id` (uuid FK), `name` (text), `source` (universal/ai/custom), `priority` (high/medium/low), `display_order` (int), `items` (jsonb array of `{name, our_status, notes}`), `created_at`, `updated_at`
- **`competitor_dimension_status`** — per-competitor status per dimension item
  - `id` (uuid PK), `dimension_id` (uuid FK), `entity_name` (text), `item_name` (text), `status` (yes/partial/no/unknown), `source` (perplexity/manual), `evidence` (text), `last_updated`
  - **Manual overrides** (`source='manual'`) are never overwritten by Perplexity

### Legacy capability tables (DEPRECATED — pending deletion)
- `workspace_capabilities` — old flat capability list (to be dropped after 2+ weeks stable)
- `competitor_capabilities` — old per-competitor status (to be dropped)
- `entrust_capabilities` — old "our product" status (to be dropped)
- **DO NOT USE these tables in new code.** All new capability logic uses `competitive_dimensions` and `competitor_dimension_status`.

### Entity data (stored in `workspaces.categories` JSONB)
Each category: `{ name, description, focus?, entities[] }`
Each entity: `{ name, type, topic_type, priority, disambiguation_context, funding?, geo_presence?, products?, last_researched_at?, ... }`

### Other structured data tables
- `entity_products` — manually added products (checked FIRST before JSON fallback)
- `competitor_pricing` — pricing data
- `battlecards` — AI-generated battlecards
- `strategic_directions` — where heading / what means for us
- `strategic_pulse` — NOT managed by Drizzle migrations

### Key workspace fields (flat columns, not in categories JSON)
`user_perspective`, `org_description`, `org_geographies`, `tracking_types`, `competitors`, `win_factors`, `vulnerability`, `early_warning_signal`, `display_name`, `parent_workspace_id`

## Competitive Dimensions System (NEW — March 2026)

### Architecture: One source of truth, three views
1. **Settings (Product & Intelligence page)** — user defines dimensions, sets items, sets OUR status per item, sets priorities
2. **Competitor page (DimensionComparisonCard)** — Perplexity auto-researches each competitor against YOUR dimensions, shows side-by-side comparison with vulnerability alerts
3. **Capability Matrix (DimensionMatrix)** — bird's-eye view: all competitors x all dimensions in one filterable grid

### Dimension sources
- **Universal** — apply to any market (e.g., Pricing & commercial model)
- **AI-suggested** — Claude generates based on product description during setup
- **Custom** — user adds manually anytime

### How dimensions feed Perplexity search
The `researchEntityDimensions` function in `ambientSearch.ts` builds a structured prompt:
- Groups items by dimension with `[HIGH/MEDIUM/LOW PRIORITY]` labels
- Includes entity disambiguation context and category focus
- Includes product context for comparison framing
- Asks Perplexity for `{status, evidence}` per item
- Skips `competitor_dimension_status` rows where `source='manual'` (user overrides persist)

### API Routes for Dimensions
| Method | Route | Auth | Purpose |
|---|---|---|---|
| GET | `/api/dimensions` | requireAuth | All dimensions for workspace |
| POST | `/api/dimensions` | requireSubAdmin | Create dimension |
| PUT | `/api/dimensions/:id` | requireSubAdmin | Update dimension |
| DELETE | `/api/dimensions/:id` | requireAdmin | Delete dimension + statuses |

## Ambient Search / Perplexity Cron

- **Schedule**: 6AM UTC daily (`server/index.ts` line ~138)
- **Trigger manually**: `curl -X POST http://localhost:5000/api/search/run-ambient`
- **Flow**: `runAmbientSearchForAllTenants()` → `runAmbientSearchForUser()` → loops `category.entities`
- **Two research functions per competitor**:
  1. `researchEntity` — funding, geo presence, products (runs if `last_researched_at` > 30 days)
  2. `researchEntityDimensions` — evaluates competitor against user's dimensions (runs if most recent status > 7 days)
- **Category context**: `categoryFocus = category.focus || category.description || undefined`
- **Research prompt uses**: `entity.name` + `entity.disambiguation_context` + `category.focus || category.description`
- **Perplexity quota**: topped up March 14, 2026
- **Rate limiter**: `RateLimiter(10, 60000)` — 10 calls per minute
- **JSON parsing**: strips markdown code fences before `JSON.parse`
- **Entities**: 27 total (12 competitors, 5 regulations, 9 topics + IDEMIA)
- **Manual override protection**: `competitor_dimension_status` rows with `source='manual'` are never overwritten

## Products & Solutions Data Flow

Two data sources, checked in priority order:
1. **`entity_products` DB table** — manually added or saved products (checked first)
2. **`entity.products` JSON** — Perplexity auto-researched fallback (if DB table empty)

Perplexity-sourced products have synthetic IDs like `perplexity-0`. The `ProductsCard` handles this:
- Edit → converts PUT to POST (creates real DB record)
- Delete → calls `POST /api/entities/:id/products/mark-managed` to clear JSON fallback, then saves remaining as real DB records

## Known Patterns & Gotchas

### Always do this
```typescript
// Use db.execute for INSERT/UPDATE
await db.execute(sql`UPDATE workspaces SET categories = ${JSON.stringify(cats)} WHERE id = ${id}`);

// Add staleTime to queries needing fresh data
useQuery({ queryKey: [...], staleTime: 0, gcTime: 0, refetchOnMount: "always" })

// Use apiRequest for authenticated fetches in frontend
const res = await apiRequest("GET", "/api/dimensions");

// Add || 'admin' role fallback
const role = profileResult.rows[0]?.role || 'admin';

// Use parseArr() for PostgreSQL array fields returning as strings
const arr = parseArr(row.some_array_field);

// For IN clauses with Drizzle, use sql.join (NOT raw arrays)
WHERE dimension_id IN (${sql.join(ids.map(id => sql`${id}::uuid`), sql`, `)})

// Manual overrides must persist — always check source before overwriting
if (existingRow.source === 'manual') { /* skip Perplexity update */ }
```

### Never do this
```typescript
// Never use pool.query for mutations
pool.query('UPDATE workspaces...')  // ❌

// Never define form components inside parent components
function ParentComponent() {
  function FormComponent() { ... }  // ❌
}

// Never run drizzle migrations on strategic_pulse table
drizzle-kit push  // ❌ will drop columns

// Never use raw fetch() for authenticated API calls in frontend
fetch("/api/dimensions").then(r => r.json())  // ❌ no auth headers
// Use apiRequest("GET", "/api/dimensions") instead ✅

// Never pass JS arrays directly in Drizzle sql template
WHERE id = ANY(${jsArray}::uuid[])  // ❌ Drizzle can't handle this

// Never overwrite manual status overrides from Perplexity
// Always check source='manual' before updating competitor_dimension_status
```
